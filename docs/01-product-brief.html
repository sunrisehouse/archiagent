Problem

문제 정의 — 산출물이 "문서"라서 생기는 일

#문제증상
P1문서가 실제와 안 맞음시스템은 계속 배포·변경되는데 문서는 갱신이 늦음 → 설계와 실제의 괴리(drift)
P2변경 영향 범위를 모름컴포넌트 간 의존성·영향 범위를 문서에서 추적 불가. "이거 바꾸면 뭐가 깨지지?"에 답 못함
P3결정 이유가 안 남음설계 의도(why)는 문서에, 실제 운영 상태(what-is)는 인프라 콘솔에. 서로 연결 안 됨
P4자연어로 못 물어봄"현재 결제 도메인 구조 알려줘" 같은 자연어 질의·온보딩 설명을 매번 수작업

근본 원인 — 산출물이 연결된 구조(graph)가 아니라 납작한 텍스트(doc)로 저장되고, 실제 시스템 상태와 분리되어 있다.

Why archiagent

기존 문서 산출물 vs archiagent

관점기존 문서 산출물 · Confluence·PPT·정적 다이어그램archiagent
최신성 P1✕ 작성 시점에 고정 → 금방 낡음✓ 실제 배포와 대조해 drift 자동 탐지
추적성 P2✕ 납작한 텍스트, 의존성 수동 추적✓ 그래프 탐색으로 다단계 영향 분석
의도+실제 P3✕ why는 문서, what은 콘솔 — 분리✓ 두 에이전트가 교차 결합
질의·온보딩 P4✕ 사람이 매번 읽고 해석✓ 자연어 질의로 즉시 응답

문서는 한 번 찍는 스냅샷, archiagent는 살아있고 질의 가능한 구조.

Intent

기획 의도

① 문서 → 지식 그래프

컴포넌트·의존성·결정(ADR)을 노드·엣지로(Graph DB) + 자연어 의미검색(Vector DB). → P2·P4

② 설계 ↔ 실제 연계

설계를 아는 Agent A + 배포 상태를 아는 Agent B를 오케스트레이터로 묶어 "의도 vs 실제"를 교차 질의. → P1·P3

③ 반복 작업 보조

영향분석 · 현황 파악 · 온보딩 설명 같은 아키텍트의 반복 업무를 에이전트가 대신.

④ 시나리오 중심 PoC

완성형 제품이 아니라, 몇 가지 시나리오로 에이전트 간 연계의 가치를 시연.

주요 기능 · 시나리오 개요

문제별 시나리오 매핑

문제시나리오한 줄 요약
P1 문서가 실제와 안 맞음① 설계 vs 실제 Drift 탐지설계서와 배포 현실의 차이를 즉시 리포트
P2 변경 영향 범위를 모름② 변경 영향도 분석"바꾸면 어디가 깨지나" + 실제 가동 여부로 우선순위
P3 결정 이유가 안 남음③ 의사결정 지원과거 결정(why) + 현재 실측(what)을 한 화면에
P4 자연어로 못 물어봄④ 아키텍처 Q&A / 온보딩자연어로 현재 구조를 설명

각 시나리오는 Agent A(설계) + Agent B(실제)를 오케스트레이터가 조합해야만 풀리는, 단일 에이전트로는 불가능한 작업이다.

Wrap-up

PoC 성공 기준

  • 4개 시나리오 각각에서 오케스트레이터가 두 에이전트를 호출·조합해 기대 산출 생성
  • Drift 리포트가 누락 / 추가 / 불일치 세 범주를 구분
  • 영향도 분석이 그래프 다단계 의존성을 따라가고, 실제 가동 여부로 우선순위화
  • Agent B는 모킹된 System State Contract 응답으로 대체 가능(외부 스택 미정 대응)
