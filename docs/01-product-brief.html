① 현황 · SI 산출물

SI 생애주기 × 아키텍트 산출물

분석→ 설계→ 개발→ 테스트→ 이행
단계입력아키텍트가 만드는 산출물(문서)
분석고객 인터뷰 · RFP요구사항 정의서 · 현행 시스템 분석서
설계분석 산출물아키텍처 정의서 · 아키텍처 설계서 (활동: 개발 인프라 구성)
개발설계서문서 산출물 없음 — 개발자 지원 · 설계 준수 점검
테스트구현체가용성 테스트 결과서 · 성능 테스트 결과서
이행전체 산출물인수인계 자료

아키텍트는 SI 전 단계에 걸쳐 이런 문서를 만든다. 그런데 이 산출물이 '문서'라서 — 다음 문제가 생긴다.

② 문제점

문서라서 — 변경·탐색에 약하다

변경 영향을 모름

한 컴포넌트를 바꾸면 어디까지 고쳐야 하나? 문서엔 의존 관계가 흩어져 있어 끝까지 추적 못 함.

찾기가 어려움

"이 부분 지금 어떻게 돼 있지?" 궁금할 때마다 여러 문서를 사람이 직접 뒤져야 함.

감리 추적이 안 됨

요구사항 정의서의 항목이 어느 설계서에 반영되고 실제로 구축됐는지 — 증빙을 짚기 어렵다.

설계 변경 연쇄

개발·인프라 중 용량 한계(노드 수·디스크↑)로 설계부터 바꿔야 할 때, 무엇을 어디까지 고칠지 막막.

근본 원인 — 산출물이 연결된 구조(graph)가 아니라 납작한 텍스트(doc)로 저장되고, 실제 시스템 상태와 분리되어 있다.

① 문서 → 지식 그래프

산출물을 Artifact 노드 + 의존성·추적성(요구→설계→구축) 엣지로. 자연어 의미검색까지.

② 설계 ↔ 실제 교차

설계를 아는 Agent A + 배포·용량을 아는 Agent B를 오케스트레이터가 묶어 "의도 vs 실제"를 교차 질의.

③ 자연어로 질의

변경 영향·감리 추적·현황 설명을 물어보면 답하는 형태로.

③ 해결방안 · 적재

산출물이 그래프가 되는 법 — 문서를 그래프로

sequenceDiagram
  actor U as Architect
  participant O as Orchestrator
  participant A as Agent A
  participant K as Graph + Vector DB
  U->>O: "아키텍처 설계서 적재해줘" + 문서
  O->>A: kb_artifact_ingest(설계서)
  A->>A: LLM 추출 — Artifact·엔티티·관계
  A-->>O: 추출 미리보기(노드/엣지)
  O->>U: 적재 확인? (AskUserQuestion)
  U-->>O: 승인
  A->>K: upsert(Artifact + DERIVED_FROM/DEFINES)
  A-->>O: 추적성 연결 완료
  O-->>U: "설계서 → 요구사항 추적성 연결됨"

핵심 문서를 통째로 넣지 않고, 의미 단위로 쪼개 연결한다.

생성되는 추적성(예시)

  • 아키텍처 설계서 —DERIVED_FROM→ 요구사항 정의서
  • 아키텍처 설계서 —DEFINES→ OrderService · ADR-009
  • 성능 테스트 결과서 —VALIDATES→ OrderService

이 추적성이 아래 시나리오 전부의 토대가 된다.

④ 시나리오 1 · 감리 대응

요구사항 추적성 — 요구 → 설계 → 구축

sequenceDiagram
  actor U as Architect
  participant O as Orchestrator
  participant A as Agent A
  participant B as Agent B
  U->>O: "요구사항 R-12, 설계·구축 어디 반영됐나?"
  O->>A: kb_graph_query(R-12 추적성)
  A-->>O: R-12 →DEFINES→ 설계서§4.2 → OrderService
  O->>B: state_get_component(OrderService)
  B-->>O: 실제 배포됨 · live
  O-->>U: 요구→설계→구축 추적 체인 + 검증 링크

상황 감리 — 요구사항이 실제로 반영·구축됐는지 증빙 필요.

산출 — 추적 체인(예시)

  • 요구사항 정의서 R-12 (분석)
  • → 아키텍처 설계서 §4.2 (설계 · DERIVED_FROM)
  • → OrderService 구축·live (Agent B 확인)
  • → 성능 테스트 결과서 VALIDATES (검증)

"어디에 반영됐나"를 클릭 한 번으로.

④ 시나리오 2 · 설계 변경

설계 변경 영향도 — 용량 한계에서 시작

sequenceDiagram
  actor U as Architect
  participant O as Orchestrator
  participant A as Agent A
  participant B as Agent B
  U->>O: "OrderDB 디스크 한계 — 샤딩 추가하면 설계 어디부터?"
  O->>B: state_get_component(OrderDB)
  B-->>O: 디스크 92% · 증설/샤딩 필요
  O->>A: 역방향 DEFINES* (OrderDB 관련 설계·서비스)
  A-->>O: 설계서§5 데이터계층 · OrderService · ADR-021
  O-->>U: 수정 대상 설계서 + 영향 서비스 + 갱신 체크리스트

상황 개발·인프라 중 용량 한계 → 설계부터 변경해야 함.

산출 — 변경 가이드(예시)

  • 먼저 고칠 설계 아키텍처 설계서 §5 (데이터 계층)
  • 영향 서비스 OrderService · PaymentService
  • 연쇄 갱신 ADR-021 · 용량 산정서 · 배포 토폴로지

실제 용량(Agent B) + 설계 추적성(Agent A)을 합쳐야 풀린다.

④ 시나리오 3 · 최신성

설계 vs 실제 Drift 탐지

sequenceDiagram
  actor U as Architect
  participant O as Orchestrator
  participant A as Agent A
  participant B as Agent B
  U->>O: "결제 도메인 설계 대비 실제 차이?"
  O->>A: kb_graph_query(payment 설계)
  A-->>O: PaymentSvc→FraudCheck·LedgerSvc
  O->>B: state_get_topology(payment)
  B-->>O: FraudCheck 없음 · PayoutWorker 추가
  O->>O: drift_compare()
  O-->>U: Drift 리포트

상황 분기 아키텍처 리뷰. 설계서는 6개월 전 작성.

산출 — Drift 리포트(예시)

  • 누락 FraudCheck — 설계O / 실제X
  • 추가 PayoutWorker — 실제O / 설계X
  • 불일치 LedgerService v1.0 → v1.1

설계서와 배포 현실의 차이를 즉시 리포트.

④ 시나리오 4 · 온보딩

아키텍처 Q&A / 온보딩

sequenceDiagram
  actor U as Newcomer
  participant O as Orchestrator
  participant A as Agent A
  participant B as Agent B
  U->>O: "주문 시스템 구조 설명해줘"

산출(예시)

  • 설계 의도(why) + 실제 상태(what) 통합 설명

  • 관련 ADR · 소유 팀까지 한 번에
