# 산출물 ① — Agent 기획 의도 및 주요 기능

> 프로젝트: **archiagent** — 소프트웨어 아키텍트를 돕는 에이전트
> 본 문서는 대회 제출용 기획 산출물의 초안이다. (자매 문서: `02-architecture.md`)

---

## 1. 문제 정의 (Problem)

아키텍처 산출물(설계서, 분석서, ADR 등)은 대부분 **정적 문서**(Word/PPT/Wiki)로 관리된다. 이 방식은 다음 문제를 낳는다.

| # | 문제 | 증상 |
|---|------|------|
| P1 | **최신성 붕괴** | 시스템은 계속 배포·변경되는데 문서는 갱신이 늦거나 안 됨 → 설계와 실제의 **괴리(drift)** |
| P2 | **추적 불가** | 컴포넌트 간 의존성·영향 범위를 문서에서 추적 불가. "이거 바꾸면 뭐가 깨지지?"에 답 못함 |
| P3 | **의도-실제 단절** | 설계 의도(why)는 문서에, 실제 운영 상태(what-is)는 인프라 콘솔에. 서로 연결되지 않음 |
| P4 | **질의 불가** | "현재 결제 도메인 구조 알려줘" 같은 자연어 질의·온보딩 설명을 사람이 매번 수작업 |

**근본 원인:** 산출물이 *연결된 구조(graph)* 가 아니라 *납작한 텍스트(doc)* 로 저장되고, *실제 시스템 상태* 와 분리되어 있다.

---

## 2. 기획 의도 (Intent)

1. **산출물을 문서가 아닌 지식 그래프로** — 컴포넌트·의존성·결정(ADR)을 노드·엣지로 구조화(Graph DB)하고, 자연어 의미검색을 위해 Vector DB를 병행한다. → P2, P4 해소
2. **설계(의도)와 실제(배포)를 연계** — 설계 지식을 가진 **Agent A** 와, 실제 배포 상태를 아는 **Agent B** 를 오케스트레이터로 묶어 "의도 vs 실제"를 한 자리에서 교차 질의한다. → P1, P3 해소
3. **아키텍트의 반복 작업을 보조** — 영향분석, 현황 파악, 온보딩 설명 같은 반복 업무를 에이전트가 대신한다.
4. **완성형 제품이 아니라 연계 가치를 보여주는 PoC** — 몇 가지 시나리오로 **에이전트 간 연계**의 효용을 시연하는 것이 1차 목표다.

---

## 3. 시스템 개요 (구성 요소)

```
[Architect] → [Orchestrator] ─┬─→ Agent A (Architecture Knowledge Agent)  ── Graph/Vector DB
                              └─→ Agent B (System State Agent, 외부)        ── Cloud/Infra
```

| 구성 | 역할 | 비고 |
|------|------|------|
| **Agent A — 아키텍처 지식 에이전트** | 산출물을 Graph DB + Vector DB로 관리·질의·갱신 | **우리가 만드는 핵심** |
| **Agent B — 시스템 상태 에이전트** | 클라우드/인프라에 배포된 **실제 현재 상태** 조회·리포트 | **별도 팀이 별도 스택으로 제작.** 본 산출물에서는 추상 인터페이스(MCP 계약)로 연계 |
| **Orchestrator** | 사용자 의도 분해 → 두 에이전트 호출·조합 → 답변 합성 | Claude Agent SDK 기반 |

> Agent B의 기술 스택은 미정이므로, 본 설계는 Agent B를 **"시스템 상태 질의 계약(System State Contract)"** 을 만족하는 블랙박스 협력자로 취급한다. (계약 상세는 `02-architecture.md` 참고)

---

## 4. 주요 기능 — 연계 시나리오 3종

각 시나리오는 **Agent A(설계 의도) + Agent B(실제 상태)** 를 오케스트레이터가 조합해야만 풀리는, 단일 에이전트로는 불가능한 작업을 보여준다.

### 시나리오 1 — 설계 vs 실제 Drift 탐지 *(연계의 핵심 가치)*
- **트리거:** "결제 도메인 설계 대비 실제 배포 차이 알려줘"
- **흐름:** A가 *설계상* 결제 토폴로지 제공 → B가 *실제 배포된* 결제 컴포넌트 제공 → Orchestrator가 교차 비교
- **산출:** Drift 리포트 — 설계엔 있으나 미배포 / 배포됐으나 설계에 없음 / 버전·의존성 불일치
- **해소 문제:** P1, P3

### 시나리오 2 — 변경 영향도 분석
- **트리거:** "Auth 서비스 인터페이스를 바꾸면 영향 범위는?"
- **흐름:** A가 그래프 역방향 의존성 탐색(`DEPENDS_ON*`)으로 영향 후보 도출 → B가 그 후보들의 실제 가동·트래픽 여부 확인 → Orchestrator가 우선순위화
- **산출:** 영향 받는 서비스/ADR 목록 + "실제 운영 중인 것" 우선 표기
- **해소 문제:** P2, P3

### 시나리오 3 — 아키텍처 Q&A / 온보딩
- **트리거:** "현재 주문 시스템 구조 설명해줘" (신규 입사자)
- **흐름:** A가 주문 도메인 서브그래프 + 관련 ADR(why) 제공 → B가 실제 배포 토폴로지·헬스 제공 → Orchestrator가 통합 설명 생성
- **산출:** 의도(why) + 실제(what) 통합 설명 + 그래프 뷰
- **해소 문제:** P3, P4

> (확장 후보) 시나리오 4 — 산출물 자동 동기화: B가 신규 배포 컴포넌트 감지 → A가 그래프 산출물 갱신. 이번 PoC 범위에서는 데이터 파이프라인의 *Reconcile* 단계로만 설계에 포함한다.

---

## 5. 성공 기준 (PoC)

- [ ] 3개 시나리오 각각에 대해, 오케스트레이터가 두 에이전트를 호출·조합해 기대 산출을 생성한다.
- [ ] Drift 리포트가 "설계에만 있음 / 실제에만 있음 / 불일치" 세 범주를 구분한다.
- [ ] 영향도 분석이 그래프 다단계 의존성을 따라가고, 실제 가동 여부로 우선순위를 매긴다.
- [ ] Agent B는 모킹된 **System State Contract** 응답으로 대체 가능하다(외부 스택 미정 대응).
