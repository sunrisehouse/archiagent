## 2. 1 · 소개 — SI 프로젝트의 단계별 산출물

- **목차 제목:** 1 · Introduction · 소개
- **거버닝 메시지:** 아키텍트는 SI 전 단계에 걸쳐 이런 문서를 만든다. 그런데 이 산출물이 **'문서'**라서 — 다음 문제가 생긴다.
- **내부 컨텐츠:**
  - 단계 띠: 분석 → 설계 → 개발 → 테스트 → 이행
  - 표 (단계 | 입력 | 아키텍트가 만드는 산출물(문서)):
    - 분석 | 고객 인터뷰·RFP | 요구사항 정의서 · 현행 시스템 분석서
    - 설계 | 분석 산출물 | 아키텍처 정의서 · 아키텍처 설계서 (활동: 개발 인프라 구성)
    - 개발 | 설계서 | 문서 산출물 없음 — 개발자 지원·설계 준수 점검
    - 테스트 | 구현체 | 가용성 테스트 결과서 · 성능 테스트 결과서
    - 이행 | 전체 산출물 | 인수인계 자료

## 3. 2 · 문제점 — 문서라서 만들기도, 고치기도, 찾기도 어렵다

- **목차 제목:** 2 · 문제점
- **거버닝 메시지:** 근본 원인 — 산출물이 **연결된 구조(graph)**가 아니라 **납작한 텍스트(doc)**로 저장되고, **실제 시스템 상태**와 분리되어 있다.
- **내부 컨텐츠:** (경고 카드)
  - **문서 작성부터 노동집약적** — 단계마다 문서를 손으로 작성, 바뀌면 여러 문서를 다시 고쳐 씀. 문서 간 정합성도 전부 수작업.
  - **변경 영향을 모름** — 한 컴포넌트를 바꾸면 어디까지 고쳐야 하나? 의존 관계가 흩어져 끝까지 추적 못 함.
  - **찾기가 어려움** — "지금 어떻게 돼 있지?" 궁금할 때마다 여러 문서를 직접 뒤져야 함.
  - **감리 추적이 안 됨** — 요구 항목이 어느 설계서에 반영되고 실제 구축됐는지 증빙을 짚기 어렵다.
  - **설계 변경 연쇄** — 용량 한계(노드 수·디스크↑)로 설계부터 바꿔야 할 때, 무엇을 어디까지 고칠지 막막.

## 4. 3 · 해결방안 (개요) — 산출물을 살아있는 지식 그래프로

- **목차 제목:** 3 · 해결방안
- **거버닝 메시지:** 산출물을 살아있는 지식 그래프로 — 작성하며 채우고, 기존 문서는 적재.
- **내부 컨텐츠:**
  - > 다이어그램: Architect → Orchestrator(의도 분해·라우팅·합성) → Agent A(아키텍처 지식, subagent) + Agent B(시스템 상태, MCP tool) → Graph DB+Vector DB / Cloud·Infra
  - 카드 3:
    - **1. 작성·적재로 그래프 구축** — 신규는 인터뷰로 작성하며 채우고 기존 문서는 적재. 어느 쪽이든 Artifact 노드·추적성(요구→설계→구축)으로 연결.
    - **2. 설계 ↔ 실제 교차** — 설계를 아는 Agent A + 배포·용량을 아는 Agent B를 오케스트레이터가 묶어 "의도 vs 실제" 교차 질의.
    - **3. 자연어로 질의** — 변경 영향·감리 추적·현황 설명을 물어보면 답하는 형태로.

## 5. 3 · 해결방안 · 토대 — 산출물이 그래프가 되는 법

- **목차 제목:** 3 · 해결방안 · 토대
- **거버닝 메시지:** 어느 경로든 같은 Artifact·추적성으로 — 이 추적성이 모든 시나리오의 토대.
- **내부 컨텐츠:**
  - > 다이어그램(시퀀스): "설계서 적재해줘"+문서 → kb_artifact_ingest → LLM 추출(Artifact·엔티티·관계) → 추출 미리보기 → 적재 확인(AskUserQuestion) → 승인 → upsert(Artifact + DERIVED_FROM/DEFINES) → 추적성 연결 완료
  - **두 경로 — 결과는 같은 그래프**
    - 1) 작성: 신규 — 인터뷰로 문서를 만들며 그래프에 적재 (시나리오 1)
    - 2) 적재: 기존 문서를 의미 단위로 쪼개 연결
  - **생성되는 추적성(예시):** 설계서 —DERIVED_FROM→ 요구사항 정의서 / 설계서 —DEFINES→ OrderService·ADR-009 / 성능 테스트 결과서 —VALIDATES→ OrderService

## 6. 4 · 주요 기능 — 에이전트가 하는 9가지

- **목차 제목:** 4 · 기능 · 에이전트가 할 수 있는 것
- **거버닝 메시지:** 다음 장부터 이 기능들을 **시나리오로 엮어** 시연.
- **내부 컨텐츠:** 표 (기능 | 하는 일)
  - 산출물 작성·갱신 | 자연어로 요구·현행·설계 초안 작성, 단계별 그래프 누적, As-Is 제약 검증·경고
  - 산출물 적재 | 기존 문서를 Artifact·엔티티·추적성으로 그래프화
  - 추적성 탐색 | 요구↔설계↔구축↔검증 양방향 추적
  - 변경 영향도 분석 | 역방향 의존 추적 + 영향 범위·갱신 체크리스트
  - 자연어 질의·설명 | 의미검색 → 그래프, 의도(why)+실제(what) 통합 설명
  - 실제 시스템 연계 | 실제 배포·용량·헬스 확인 (외부 Agent B)
  - Drift 비교 | 설계 vs 실제 차이 — 누락·추가·불일치
  - 사람 확인 게이트 | 비가역 작업(그래프 쓰기·산출물 확정) 전 승인
  - 그래프 시각화 | 서브그래프·의존성 뷰로 시각적 이해

## 7. 5 · 시나리오 한눈에

- **목차 제목:** 5 · 시나리오 한눈에
- **거버닝 메시지:** 공통 — 모두 **작성하며 쌓인 그래프** 위에서 동작. 다음 장부터 하나씩 자세히.
- **내부 컨텐츠:** 표 (시나리오 | 이런 상황에서 | 이렇게 물으면 | 이런 걸 받는다)
  - 1 · 문서 작성 | 신규 SI, 백지에서 | "AsIs처럼 노드 대수 유지해서 설계서까지 써줘" | 요구·현행·설계서 초안 + 추적성 자동 연결
  - 2 · 감리 대응 | 감리 증빙 필요 | "요구 R-12, 어디에 반영됐어?" | 요구→설계→구축 추적 체인
  - 3 · 설계 변경 | 용량 한계 도달 | "샤딩 추가하면 설계 뭐부터 고쳐?" | 변경 대상 설계 + 영향도 체크리스트
  - 4 · Drift 탐지 | 설계서가 오래됨 | "설계랑 실제 배포 차이 알려줘" | 누락·추가·불일치 리포트
  - 5 · Q&A/온보딩 | 신규 입사자 | "이 시스템 구조 설명해줘" | 설계 의도+실제 상태 통합 설명 + 그래프 뷰

## 8. 5.1 · 문서 작성 — 요구 → 현행 → 설계, 단계마다 그래프에 쌓으며

- **목차 제목:** 5.1 · 문서 작성
- **거버닝 메시지:** 각 산출물이 그래프에 **쌓여 다음 단계의 입력**이 된다 — 문서가 아니라 연결된 그래프라서 가능. 3단계에서 자연어 제약(노드 대수)을 어기면 **경고**. 사람은 "작성"이 아니라 "검토"만.
- **내부 컨텐츠:**
  - 💬 질의: "인터뷰·RFP로 요구사항 잡고 → 현행 시스템 분석하고 → ToBe는 AsIs처럼 노드 대수 유지해서 설계서까지"
  - 파이프라인 (단계마다 그래프 적재 → 다음 단계 입력):
    - [분석] 1. 요구사항 정의서 — 고객 인터뷰+RFP → 요구 항목 R-1..n 작성 → 그래프 적재
    - [분석] 2. 현행 시스템 분석서 — 현행 문서+인터뷰 → As-Is 노드 대수·토폴로지·용량 작성 → 그래프 적재
    - [설계] 3. 아키텍처 설계서 — 그래프(현행+요구) + 자연어 "AsIs처럼 노드 대수 유지" → ToBe 설계서 초안 → 요구→설계 추적성 자동

## 9. 5.2 · 감리 대응 — 요구사항 추적성 (요구 → 설계 → 구축)

- **목차 제목:** 5.2 · 감리 대응
- **거버닝 메시지:** "어디에 반영됐나"를 사람이 문서 뒤지지 않고 한 번에.
- **내부 컨텐츠:**
  - 💬 질의: "요구사항 R-12, 설계랑 구축 어디에 반영됐어?"
  - 3단계: 1) 묻는다(요구사항 번호 하나) → 2) 에이전트가 한다(요구→설계→서비스 추적성 따라가고 실제 운영 확인) → 3) 받는다(추적 체인 + 증빙)
  - 결과(예시): 요구 R-12 → 아키텍처 설계서 §4.2 → OrderService(구축·운영 중) → 성능 테스트 통과

## 10. 5.3 · 설계 변경 — 설계 변경 영향도 (용량 한계에서 시작)

- **목차 제목:** 5.3 · 설계 변경
- **거버닝 메시지:** 실제 용량 + 설계 추적성을 합쳐야 풀린다.
- **내부 컨텐츠:**
  - 💬 질의: "OrderDB 디스크가 꽉 차가는데, 샤딩 추가하면 설계 뭐부터 고쳐?"
  - 3단계: 1) 묻는다(용량 한계 상황) → 2) 에이전트가 한다(실제 용량 확인 + 엮인 설계서·서비스 역방향 추적) → 3) 받는다(변경 가이드)
  - 결과(예시): 먼저 고칠 설계 — 아키텍처 설계서 §5(데이터 계층), 디스크 92% / 영향 서비스 — OrderService·PaymentService / 연쇄 갱신 — ADR-021·용량 산정서·배포 토폴로지

## 11. 5.4 · 최신성 — 설계 vs 실제 Drift 탐지

- **목차 제목:** 5.4 · 최신성
- **거버닝 메시지:** 6개월 된 설계서와 배포 현실의 차이를 즉시.
- **내부 컨텐츠:**
  - 💬 질의: "결제 도메인, 설계랑 실제 배포가 얼마나 달라?"
  - 3단계: 1) 묻는다(도메인 하나) → 2) 에이전트가 한다(설계 vs 실제 배포 나란히 비교) → 3) 받는다(Drift 리포트: 누락·추가·불일치)
  - 결과(예시): 누락 FraudCheck(설계엔 있는데 실제 없음) / 추가 PayoutWorker(실제엔 있는데 설계 없음) / 불일치 LedgerService v1.0 → v1.1

## 12. 5.5 · 온보딩 — 아키텍처 Q&A / 온보딩

- **목차 제목:** 5.5 · 온보딩
- **거버닝 메시지:** 문서 더미 대신 자연어로 현재 구조를 설명.
- **내부 컨텐츠:**
  - 💬 질의: "주문 시스템 구조 좀 설명해줘" (신규 입사자)
