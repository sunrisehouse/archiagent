## 1

- **목차 제목:** 산출물 ① · Product Brief
- **거버닝 메시지:** archiagent는 아키텍처 산출물을 문서가 아니라 살아있는 지식 그래프로 다룬다. 설계 의도와 실제 배포를 잇는 멀티 에이전트다.
- **내부 컨텐츠:**
  - 제목: **기획 의도 및 주요 기능**

## 2

- **목차 제목:** 1 · Introduction · 소개
- **거버닝 메시지:** 아키텍트는 SI 전 단계에 걸쳐 이런 문서를 만든다.
- **내부 컨텐츠:**
  - 단계 띠: 분석 → 설계 → 개발 → 테스트 → 이행
  - 표 (단계 | 입력 | 아키텍트가 만드는 산출물(문서)):
    - 분석 | 고객 인터뷰·RFP | 요구사항 정의서 · 현행 시스템 분석서
    - 설계 | 분석 산출물 | 아키텍처 정의서 · 아키텍처 설계서 (활동: 개발 인프라 구성)
    - 개발 | 설계서 | 문서 산출물 없음 — 개발자 지원·설계 준수 점검
    - 테스트 | 구현체 | 가용성 테스트 결과서 · 성능 테스트 결과서
    - 이행 | 전체 산출물 | 인수인계 자료

## 3

- **목차 제목:** 2 · 문제점
- **거버닝 메시지:** 근본 원인은 산출물이 연결된 구조인 그래프가 아니라 납작한 텍스트인 문서로 저장되고, 실제 시스템 상태와 분리되어 있다는 점이다.
- **내부 컨텐츠:** (경고 카드)
  - **문서 작성부터 힘듦** — 단계마다 문서를 손으로 작성, 바뀌면 여러 문서를 다시 고쳐 씀. 문서 간 정합성도 전부 수작업.
  - **변경 영향을 모름** — 한 컴포넌트를 바꾸면 어디까지 고쳐야 하나? 의존 관계가 흩어져 끝까지 추적 못 함.
  - **찾기가 어려움** — "지금 어떻게 돼 있지?" 궁금할 때마다 여러 문서를 직접 뒤져야 함.
  - **감리 추적이 안 됨** — 요구 항목이 어느 설계서에 반영되고 실제 구축됐는지 증빙을 짚기 어렵다.
  - **설계 변경 연쇄** — 용량 한계(노드 수·디스크↑)로 설계부터 바꿔야 할 때, 무엇을 어디까지 고칠지 막막.

## 4

- **목차 제목:** 3.1 · 해결방안 · 지식 에이전트
- **거버닝 메시지:** 첫 번째 에이전트가 산출물인 설계 지식을 그래프 DB에 노드와 관계로 쌓아 관리한다. 새로 작성하면서 채우고, 기존 문서는 적재한다.
- **내부 컨텐츠:**
  - > 다이어그램: Architect → Orchestrator(의도 분해·라우팅·합성) → Agent A(아키텍처 지식 에이전트, subagent) → Graph DB + Vector DB
  - 하는 일:
    - **작성·적재로 그래프 구축** — 신규는 인터뷰로 작성하며 채우고 기존 문서는 적재. 어느 쪽이든 Artifact 노드·추적성(요구→설계→구축)으로 연결.
    - **자연어로 질의·설명** — 추적성·감리·현황 설명을 물어보면 답하는 형태로.
  - 핵심: 흩어진 문서가 아니라 **연결된 그래프** → 작성·변경·탐색이 쉬워진다.

## 5

- **목차 제목:** 3.2 · 해결방안 · 인프라 에이전트
- **거버닝 메시지:** 두 번째 에이전트인 인프라 에이전트가 현재 클라우드 자원과 인프라 상태를 조회한다. 설계 그래프와 실제를 비교해서 변경 영향을 모르는 문제와 설계 변경 연쇄 문제를 푼다.
- **내부 컨텐츠:**
  - > 다이어그램: Orchestrator -. MCP tool .-> Agent B(인프라 에이전트) → Cloud·Infra(실제 배포·용량·헬스)
  - 하는 일:
    - **자원·인프라 상태 조회** — 실제 배포·용량·헬스를 조회 (state_* 도구, 외부 Agent B)
    - **설계 ↔ 실제 비교** — 그래프의 설계 의도(why)와 실제 자원 상태(what)를 교차
  - 이렇게 문제점을 푼다 (2번 슬라이드 연결):
    - **변경 영향을 모름** → 실제 자원에서 거꾸로 영향받는 설계·서비스를 추적
    - **설계 변경 연쇄** → 실제 용량 한계를 기준으로 먼저 고칠 설계 우선순위 제시
  - 목표: **두 에이전트(지식 + 인프라)를 함께** 두는 것 — 의도(설계)와 실제(자원)를 한 번에. (시나리오 3·4의 토대)

## 6

- **목차 제목:** 4.1 · 기능 · 지식 에이전트
- **거버닝 메시지:** 지식 에이전트가 그래프 위에서 작성과 적재부터 추적과 변경 영향 분석, 질의, 시각화까지 담당한다.
- **내부 컨텐츠:** 기능 (기능 | 하는 일)
  - 산출물 작성·갱신 | 자연어로 요구·현행·설계 초안 작성, 단계별 그래프 누적, As-Is 제약 검증·경고
  - 산출물 적재 | 기존 문서를 Artifact·엔티티·추적성으로 그래프화
  - 추적성 탐색 | 요구↔설계↔구축↔검증 양방향 추적
  - 변경 영향도 분석 | 역방향 의존 추적 + 영향 범위·갱신 체크리스트 (용량 한계는 인프라 에이전트와 결합)
  - 자연어 질의·설명 | 의미검색 → 그래프, 설계 의도(why) 설명
  - 사람 확인 게이트 | 비가역 작업(그래프 쓰기·산출물 확정) 전 승인
  - 그래프 시각화 | 서브그래프·의존성 뷰로 시각적 이해

## 7

- **목차 제목:** 4.2 · 기능 · 인프라 에이전트
- **거버닝 메시지:** 인프라 에이전트가 실제 자원과 상태를 조회하고, 설계와 비교해서 차이를 드러낸다.
- **내부 컨텐츠:** 기능 (기능 | 하는 일)
  - 실제 시스템 연계 | 실제 배포·용량·헬스 확인 (state_* 도구, 외부 에이전트)
  - 설계·실제 차이 비교 | 설계(지식 에이전트) vs 실제(인프라 에이전트) — 누락·추가·불일치
  - (전환) 다음 장부터 이 기능들을 **시나리오로 엮어** 시연.

## 8

- **목차 제목:** 5.1 시나리오 - 문서 작성
- **거버닝 메시지:** 분석 단계에서는 고객 인터뷰와 RFP, 현행 시스템 문서를 입력해 요구사항 정의서와 현행 시스템 분석서를 그래프 데이터로 만든다. 설계 단계에서는 앞선 데이터를 활용해 설계서를 만든다.
- **내부 컨텐츠:**
  - 상황: 신규 SI 프로젝트를 백지 상태에서 시작한다.
  - 묻는다: 💬 "인터뷰와 RFP로 요구사항을 잡고, 현행 시스템을 분석한 다음, 현행과 같은 노드 대수를 유지해서 설계서까지 만들어줘."
  - 에이전트가 한다:
    - 분석 단계에서 고객 인터뷰와 RFP를 입력받아 요구사항 정의서를 작성하고, 요구 항목들을 그래프에 적재한다.
    - 같은 분석 단계에서 현행 시스템 문서와 인터뷰를 입력받아 현행 시스템 분석서를 작성하고, 현행의 노드 대수와 토폴로지, 용량을 그래프에 적재한다.
    - 설계 단계에서 앞서 쌓인 현행과 요구사항 데이터를 활용해 설계서 초안을 작성하고, 요구사항과 설계 사이의 추적성을 자동으로 연결한다.
  - 받는다: 요구사항 정의서와 현행 시스템 분석서, 아키텍처 설계서 초안을 받는다. 사람은 직접 작성하지 않고 검토만 한다.

## 9

- **목차 제목:** 5.2 시나리오 - 감리 대응
- **거버닝 메시지:** 어디에 반영됐는지를 사람이 문서를 뒤지지 않고 한 번에 확인한다.
- **내부 컨텐츠:**
  - 상황: 감리에서 요구사항이 실제로 반영됐다는 증빙이 필요하다.
  - 묻는다: 💬 "요구사항 R-12가 설계와 구축 어디에 반영됐는지 알려줘."
  - 에이전트가 한다: 요구사항에서 설계서, 서비스로 이어지는 추적성을 따라가고, 그 서비스가 실제로 구축되어 운영 중인지 확인한다.
  - 받는다: 요구부터 검증까지 한 줄로 이어진 추적 체인을 받는다. 예를 들어 요구사항 R-12는 아키텍처 설계서 4.2절로 이어지고, OrderService로 구축되어 운영 중이며, 성능 테스트를 통과한 것으로 나타난다.

## 10

- **목차 제목:** 5.3 시나리오 - 설계 변경
- **거버닝 메시지:** 실제 인프라 변경을 설계 추적성과 합쳐야 어디부터 고칠지 알 수 있다.
- **내부 컨텐츠:**
  - 상황: 부하가 늘어 VM을 새로 추가하고 일부 서버의 IP가 바뀌어서, 설계 내용을 고쳐야 한다.
  - 묻는다: 💬 "VM을 추가하고 IP가 바뀌었는데, 설계서를 어디부터 고쳐야 해?"
  - 에이전트가 한다: 실제로 추가된 VM과 바뀐 IP를 확인하고, 그 자원과 엮인 설계서와 서비스를 역방향으로 추적해서 영향 범위를 찾는다.
  - 받는다: 먼저 고칠 설계와 영향받는 서비스, 함께 갱신할 항목을 받는다. 예를 들어 배포 토폴로지와 네트워크 구성도를 먼저 고치고, OrderService와 PaymentService가 영향받으며, 관련 ADR과 용량 산정서를 함께 갱신해야 하는 것으로 나타난다.

## 11

- **목차 제목:** 5.4 시나리오 - 최신성
- **거버닝 메시지:** 6개월 된 설계서와 배포 현실의 차이를 즉시 보여준다.
- **내부 컨텐츠:**
  - 상황: 설계서가 오래되어 실제 배포와 얼마나 달라졌는지 확인해야 한다.
  - 묻는다: 💬 "결제 도메인이 설계와 실제 배포가 얼마나 다른지 알려줘."
  - 에이전트가 한다: 설계서에 적힌 구조와 실제로 배포된 구조를 나란히 놓고 비교한다.
  - 받는다: 누락과 추가, 불일치 세 가지로 정리한 차이 리포트를 받는다. 예를 들어 설계에는 있지만 실제로 없는 FraudCheck, 실제로는 돌고 있지만 설계에 없는 PayoutWorker, 버전이 어긋난 LedgerService를 확인한다.

## 12

- **목차 제목:** 5.5 시나리오 - 온보딩
- **거버닝 메시지:** 문서 더미 대신 자연어로 현재 구조를 설명한다.
- **내부 컨텐츠:**
  - 상황: 신규 입사자가 시스템 구조를 빠르게 이해해야 한다.
  - 묻는다: 💬 "주문 시스템 구조를 설명해줘."
  - 에이전트가 한다: 왜 그렇게 설계했는지와 지금 실제로 어떻게 배포되어 있는지를 함께 끌어와서 하나로 설명한다.
  - 받는다: 설계 의도와 현재 상태를 합친 설명을 받는다. 주문 도메인 그래프와 관련 ADR, 담당 팀까지 함께 확인한다.
