# 산출물 ① 부록 — 시나리오 상세 워크스루 (콘텐츠)

> 이 문서는 `01-product-brief.md`의 시나리오를 **실제 에이전트 사용 흐름**으로 자세히 풉니다.
> 에이전트를 어떻게 켜고, 어떤 데이터를 넣고, 어떤 명령을 입력하는지를 **명령어 실행 단위**로 보여줍니다.
> 데이터는 이해를 돕기 위해 아주 단순한 형태(짧은 목록)로 구성했습니다.

---

## 공통 — 에이전트를 켜는 방법

에이전트는 터미널에서 CLI로 실행합니다. 작업 폴더에는 에이전트에 넣을 자료 파일을 미리 둡니다.

```
$ cd my-project
$ archiagent
archiagent를 시작합니다. 무엇을 도와드릴까요?
>
```

- `>` 프롬프트에 사람이 **자연어로** 요청을 입력합니다.
- 자료 파일은 요청 문장에서 **파일 이름으로 가리키면** 에이전트가 작업 폴더에서 읽습니다.
- 그래프에 쓰거나 산출물을 확정하는 작업 전에는 에이전트가 **미리보기를 보여주고 확인(y/n)** 을 받습니다. 이것이 사람 확인 게이트입니다.

아래 모든 시나리오는 같은 작업 폴더에서, 한 세션 안에서 이어집니다.

---

## 시나리오 5.1 — 산출물 작성

신규 쇼핑몰 구축 프로젝트의 산출물을 처음부터 작성합니다.

### 1) 준비하는 데이터

작업 폴더에 아래 세 파일을 둡니다. 모두 짧은 목록입니다.

`rfp.md` — 발주처가 준 요구 목록입니다.

```
# 쇼핑몰 신규 구축 RFP
- 회원 가입과 로그인 기능이 필요합니다.
- 상품을 검색하고 장바구니에 담을 수 있어야 합니다.
- 주문과 결제를 처리할 수 있어야 합니다.
- 관리자가 상품과 주문을 관리할 수 있어야 합니다.
- 동시 접속 1,000명을 견뎌야 합니다.
```

`interview.md` — 고객 인터뷰에서 나온 이야기를 다섯 건만 적었습니다.

```
# 고객 인터뷰 메모
- 결제는 카드와 간편결제를 모두 지원하면 좋겠습니다.
- 주문이 몰리는 시간대에도 느려지지 않았으면 합니다.
- 기존 회원 데이터를 그대로 옮겨오고 싶습니다.
- 화면이 모바일에서도 잘 보였으면 합니다.
- 운영팀이 직접 상품을 등록할 수 있어야 합니다.
```

`current-system.md` — 현행 시스템의 서버 구성을 짧게 적었습니다.

```
# 현행 시스템 자료
- 웹 서버 web-01 (IP 10.0.1.11, vCPU 4, 메모리 8GB, 디스크 100GB)
- 웹 서버 web-02 (IP 10.0.1.12, vCPU 4, 메모리 8GB, 디스크 100GB)
- DB 서버 db-01 (IP 10.0.2.11, vCPU 8, 메모리 16GB, 디스크 500GB)
- 현재 동시 접속은 약 300명을 처리합니다.
```

### 2) 명령을 차례대로 실행합니다

**첫째, RFP로 요구사항 정의서를 작성합니다.**

```
> rfp.md 파일을 읽고 요구사항 정의서를 작성해줘.
```

에이전트는 `rfp.md`를 읽어 요구 항목을 R-1부터 차례로 뽑고, 요구사항 정의서 초안을 만들어 보여줍니다. 그리고 그래프에 적재해도 되는지 확인을 받습니다.

```
요구사항 정의서 초안입니다.
  R-1 회원 가입과 로그인
  R-2 상품 검색과 장바구니
  R-3 주문과 결제
  R-4 관리자 상품·주문 관리
  R-5 동시 접속 1,000명 처리
이 내용을 그래프에 저장할까요? (y/n)
> y
저장했습니다. 요구사항 정의서(R-1 ~ R-5)를 그래프에 적재했습니다.
```

**둘째, 인터뷰 내용을 같은 요구사항 정의서에 더합니다.**

```
> interview.md 도 읽고 요구사항 정의서에 빠진 내용을 더해줘.
```

에이전트는 인터뷰에서 새 요구를 찾아 기존 정의서에 덧붙이고, 다시 확인을 받습니다.

```
인터뷰에서 다음 요구를 더 찾았습니다.
  R-6 카드·간편결제 지원
  R-7 피크 시간대 성능 유지
  R-8 기존 회원 데이터 이관
  R-9 모바일 화면 대응
  R-10 운영팀 상품 등록
요구사항 정의서에 추가할까요? (y/n)
> y
추가했습니다. 요구사항 정의서는 이제 R-1 ~ R-10 입니다.
```

**셋째, 현행 시스템 문서로 현행 시스템 분석서를 작성합니다.**

```
> current-system.md 를 읽고 현행 시스템 분석서를 작성해줘.
```

에이전트는 서버 목록과 사양, 현재 처리량을 정리해 현행 시스템 분석서를 만들고 그래프에 적재합니다.

```
현행 시스템 분석서를 작성했습니다.
  노드 web-01, web-02, db-01 의 IP와 사양을 정리했습니다.
  현재 동시 접속 처리량은 약 300명입니다.
그래프에 저장할까요? (y/n)
> y
저장했습니다.
```

**넷째, 요구사항과 현행 분석서를 기반으로 설계서를 작성합니다.**

```
> 요구사항 정의서와 현행 시스템 분석서를 기반으로 설계서를 작성해줘.
```

에이전트는 그래프에 쌓인 두 산출물을 함께 읽어 설계서 초안을 만들고, 각 요구가 어느 설계로 이어지는지 연결을 자동으로 만든 뒤 확인을 받습니다.

```
아키텍처 설계서 초안입니다.
  웹 계층은 web-01, web-02 를 유지하고 한 대를 더 늘려 1,000명 처리에 대비합니다. (R-5, R-7)
  결제 서비스는 카드와 간편결제를 함께 처리하도록 둡니다. (R-3, R-6)
  회원 데이터 이관 절차를 설계에 포함합니다. (R-8)
요구와 설계의 연결을 함께 저장할까요? (y/n)
> y
저장했습니다. 요구사항부터 설계까지 연결이 그래프에 만들어졌습니다.
```

### 3) 받는 결과

요구사항 정의서(R-1 ~ R-10), 현행 시스템 분석서, 아키텍처 설계서가 모두 그래프에 쌓였습니다. 세 산출물이 서로 연결되어 있어 어떤 요구가 어떤 설계로 이어지는지 바로 볼 수 있습니다.

## 시나리오 5.2 — 변경 대응

요건 하나가 바뀌었습니다. 먼저 영향 범위를 보고, 확정되면 산출물에 반영합니다.

### 1) 입력하는 변경 내용

이번에는 파일 대신 한 줄로 바로 입력합니다.

**첫째, 영향도를 먼저 물어봅니다.**

```
> 동시 접속 요건을 1,000명에서 5,000명으로 올리려고 해. 영향도가 어떻게 돼?
```

에이전트는 그래프에서 동시 접속 요건(R-5)과 연결된 산출물·설계를 따라가며 영향받는 부분을 찾아 보여줍니다. (이 단계는 조회라서 확인을 받지 않습니다.)

```
R-5(동시 접속) 변경의 영향 범위입니다.
  영향 요구: R-5, R-7(피크 성능)
  영향 설계: 웹 계층 구성, 결제 서비스 용량
  함께 볼 산출물: 현행 시스템 분석서(서버 대수)
  웹 서버를 3대에서 더 늘려야 할 가능성이 큽니다.
```

**둘째, 변경을 확정해 반영을 지시합니다.**

```
> 그 요건을 5,000명으로 확정할게. 요구사항 정의서와 설계서에 반영해줘.
```

에이전트는 요구사항 정의서의 R-5를 고치고, 연결된 설계서도 함께 고친 뒤 확인을 받습니다.

```
다음과 같이 반영하겠습니다.
  요구사항 정의서: R-5 동시 접속 1,000명 → 5,000명
  설계서: 웹 계층을 3대에서 5대로 확장, 결제 서비스 용량 상향
  변경 사유: 동시 접속 요건 상향(고객 요청)
저장할까요? (y/n)
> y
저장했습니다. 변경 내용과 사유를 그래프에 함께 기록했습니다.
```

### 2) 받는 결과

먼저 영향 범위를 받아 어디까지 손대야 하는지 파악했고, 이어서 실제로 갱신된 요구사항 정의서와 설계서를 받았습니다. 무엇을 왜 바꿨는지가 함께 남아 나중에 추적할 수 있습니다.

## 시나리오 5.3 — 설계 질의응답

설계에 관해 궁금한 점을 자연어로 물어보고 정리된 답을 받습니다.

### 1) 입력하는 질문

```
> 운영 환경 노드마다 IP랑 CPU, 메모리, 디스크 크기를 정리해줘.
```

에이전트는 그래프에 쌓인 현행 시스템 분석서와 설계서에서 노드와 사양을 찾아 한 표로 모아 보여줍니다. (조회라서 확인을 받지 않습니다.)

```
운영 환경 노드 사양입니다.

  노드      IP            CPU      메모리    디스크
  web-01    10.0.1.11     vCPU 4   8GB      100GB
  web-02    10.0.1.12     vCPU 4   8GB      100GB
  db-01     10.0.2.11     vCPU 8   16GB     500GB

  ※ 설계서에서 추가 예정인 web-03 ~ web-05 는 아직 IP가 배정되지 않았습니다.
```

### 2) 받는 결과

노드별 IP와 CPU, 메모리, 디스크 크기를 정리한 표를 한 번에 받았습니다. 여러 문서를 직접 뒤지지 않고 그래프에서 바로 모았기 때문에, 내용은 지금까지 쌓인 산출물과 일치합니다.

## 시나리오 5.4 — 산출물 출력

지금까지 작성한 산출물을 제출용 문서로 정리합니다.

### 1) 입력하는 명령

```
> 현재 산출물을 문서별로 HTML로 만들어줘.
```

에이전트는 그래프에 있는 각 산출물을 문서 단위로 모아 HTML 파일로 만들어 작업 폴더에 저장하고, 결과를 알려 줍니다.

```
다음 문서를 HTML로 만들었습니다.
  out/requirements.html   요구사항 정의서 (R-1 ~ R-10)
  out/current-system.html 현행 시스템 분석서
  out/design.html         아키텍처 설계서
각 문서의 내용은 그래프에서 가져왔기 때문에 최신 상태입니다.
```

### 2) 받는 결과

요구사항 정의서, 현행 시스템 분석서, 설계서가 각각 HTML 파일로 출력되었습니다. 앞선 시나리오에서 요건을 5,000명으로 바꾼 내용까지 반영되어, 출력물은 항상 그래프의 최신 상태와 같습니다.
