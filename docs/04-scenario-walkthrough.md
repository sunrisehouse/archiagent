# 산출물 ① 부록 — 시나리오 상세 워크스루 (콘텐츠)

> 이 문서는 `01-product-brief.md`의 시나리오를 **실제 에이전트 사용 흐름**으로 자세히 풉니다.
> 에이전트를 어떻게 켜고, 어떤 데이터를 넣고, 어떤 명령을 입력하는지를 **명령어 실행 단위**로 보여줍니다.
> 예시는 **은행 차세대 시스템 구축** 프로젝트이며, 요건은 **비기능 요건**(클라우드 전환, MSA 전환,
> 모니터링 체계 등)으로만 구성했습니다. 데이터는 이해를 돕기 위해 아주 단순한 형태(짧은 목록)로 적었습니다.

---

## 공통 — 에이전트를 켜는 방법

에이전트는 터미널에서 CLI로 실행합니다. 작업 폴더에는 에이전트에 넣을 자료 파일을 미리 둡니다.

```
$ cd bank-nextgen
$ archiagent
archiagent를 시작합니다. 무엇을 도와드릴까요?
>
```

- `>` 프롬프트에 사람이 **자연어로** 요청을 입력합니다.
- 자료 파일은 요청 문장에서 **파일 이름으로 가리키면** 에이전트가 작업 폴더에서 읽습니다.
- 그래프에 쓰거나 산출물을 확정하는 작업 전에는 에이전트가 **미리보기를 보여주고 확인(y/n)** 을 받습니다. 이것이 사람 확인 게이트입니다.

아래 모든 시나리오는 같은 작업 폴더에서, 한 세션 안에서 이어집니다.

---

## 시나리오 5.1 — 산출물 작성

은행 차세대 시스템 구축 프로젝트의 산출물을 처음부터 작성합니다.

### 1) 준비하는 데이터

작업 폴더에 아래 세 파일을 둡니다. 모두 짧은 목록입니다.

`rfp.md` — 발주처가 준 요구 목록입니다. 모두 비기능 요건입니다.

```
# 은행 차세대 시스템 구축 RFP
- 현행 온프레미스 시스템을 클라우드로 전환해야 합니다.
- 핵심 업무를 마이크로서비스(MSA) 구조로 전환해야 합니다.
- 장애를 빠르게 감지하는 모니터링 체계를 갖춰야 합니다.
- 24시간 무중단으로 안정적으로 운영되어야 합니다.
- 금융 보안 규제를 준수해야 합니다.
```

`interview.md` — 고객 인터뷰에서 나온 이야기를 다섯 건만 적었습니다.

```
# 고객 인터뷰 메모
- 피크 시간대에도 응답이 느려지지 않았으면 합니다.
- 장애가 나면 바로 알림을 받고 싶습니다.
- 특정 서비스만 따로 확장할 수 있으면 좋겠습니다.
- 배포할 때 서비스가 멈추지 않았으면 합니다.
- 감사와 규제 대응을 위해 로그가 잘 남았으면 합니다.
```

`current-system.md` — 현행 시스템의 서버 구성을 짧게 적었습니다.

```
# 현행 시스템 자료
- 업무 서버 core-01 (IP 10.0.1.21, vCPU 8, 메모리 32GB, 디스크 500GB)
- 업무 서버 core-02 (IP 10.0.1.22, vCPU 8, 메모리 32GB, 디스크 500GB)
- DB 서버 db-01 (IP 10.0.2.21, vCPU 16, 메모리 64GB, 디스크 2TB)
- 모두 자체 데이터센터(온프레미스)에서 운영합니다.
- 현재는 모놀리식 구조라서 배포할 때 전체를 함께 내렸다 올립니다.
```

### 2) 명령을 차례대로 실행합니다

**첫째, RFP로 요구사항 정의서를 작성합니다.**

```
> rfp.md 파일을 읽고 요구사항 정의서를 작성해줘.
```

에이전트는 `rfp.md`를 읽어 요구 항목을 R-1부터 차례로 뽑고, 요구사항 정의서 초안을 만들어 보여줍니다. 그리고 그래프에 적재해도 되는지 확인을 받습니다.

```
요구사항 정의서 초안입니다.
  R-1 클라우드 전환
  R-2 마이크로서비스(MSA) 전환
  R-3 장애 감지 모니터링 체계
  R-4 24시간 무중단 운영
  R-5 금융 보안 규제 준수
이 내용을 그래프에 저장할까요? (y/n)
> y
저장했습니다. 요구사항 정의서(R-1 ~ R-5)를 그래프에 적재했습니다.
```

**둘째, 인터뷰 내용을 같은 요구사항 정의서에 더합니다.**

```
> interview.md 도 읽고 요구사항 정의서에 빠진 내용을 더해줘.
```

에이전트는 인터뷰에서 새 요구를 찾아 기존 정의서에 덧붙이고, 다시 확인을 받습니다.

```
인터뷰에서 다음 요구를 더 찾았습니다.
  R-6 피크 시간대 성능 유지
  R-7 장애 발생 시 즉시 알림
  R-8 서비스별 개별 확장
  R-9 무중단 배포
  R-10 감사·규제 대응 로그 보존
요구사항 정의서에 추가할까요? (y/n)
> y
추가했습니다. 요구사항 정의서는 이제 R-1 ~ R-10 입니다.
```

**셋째, 현행 시스템 문서로 현행 시스템 분석서를 작성합니다.**

```
> current-system.md 를 읽고 현행 시스템 분석서를 작성해줘.
```

에이전트는 서버 목록과 사양, 운영 방식을 정리해 현행 시스템 분석서를 만들고 그래프에 적재합니다.

```
현행 시스템 분석서를 작성했습니다.
  노드 core-01, core-02, db-01 의 IP와 사양을 정리했습니다.
  온프레미스 데이터센터에서 모놀리식 구조로 운영 중입니다.
그래프에 저장할까요? (y/n)
> y
저장했습니다.
```

**넷째, 요구사항과 현행 분석서를 기반으로 설계서를 작성합니다.**

```
> 요구사항 정의서와 현행 시스템 분석서를 기반으로 설계서를 작성해줘.
```

에이전트는 그래프에 쌓인 두 산출물을 함께 읽어 설계서 초안을 만들고, 각 요구가 어느 설계로 이어지는지 연결을 자동으로 만든 뒤 확인을 받습니다.

```
아키텍처 설계서 초안입니다.
  현행 온프레미스 업무 서버를 클라우드로 옮깁니다. (R-1)
  핵심 업무를 계정계·여신·채널 등 마이크로서비스로 분리합니다. (R-2)
  모든 서비스에 모니터링과 장애 알림을 붙입니다. (R-3, R-7)
  서비스별로 따로 확장하고 무중단으로 배포하도록 구성합니다. (R-6, R-8, R-9)
  접근 통제와 로그 보존으로 보안 규제에 대응합니다. (R-5, R-10)
요구와 설계의 연결을 함께 저장할까요? (y/n)
> y
저장했습니다. 요구사항부터 설계까지 연결이 그래프에 만들어졌습니다.
```

### 3) 받는 결과

요구사항 정의서(R-1 ~ R-10), 현행 시스템 분석서, 아키텍처 설계서가 모두 그래프에 쌓였습니다. 세 산출물이 서로 연결되어 있어 어떤 요구가 어떤 설계로 이어지는지 바로 볼 수 있습니다.

## 시나리오 5.2 — 변경 대응

새 비기능 요건이 들어왔습니다. 먼저 영향 범위를 보고, 확정되면 산출물에 반영합니다.

### 1) 입력하는 변경 내용

이번에는 파일 대신 한 줄로 바로 입력합니다.

**첫째, 영향도를 먼저 물어봅니다.**

```
> 재해 복구를 위해 클라우드 리전을 두 곳으로 이중화하는 요건을 추가하려고 해. 영향도가 어떻게 돼?
```

에이전트는 그래프에서 이 요건과 관련된 산출물·설계를 따라가며 영향받는 부분을 찾아 보여줍니다. (이 단계는 조회라서 확인을 받지 않습니다.)

```
리전 이중화(재해 복구) 요건의 영향 범위입니다.
  영향 요구: R-1(클라우드 전환), R-4(무중단 운영)
  영향 설계: 클라우드 배포 구성, 마이크로서비스 배치, 모니터링 범위
  두 리전에 같은 서비스를 배치하고, 한쪽 장애 시 다른 리전으로 넘기는 구성이 필요합니다.
```

**둘째, 변경을 확정해 반영을 지시합니다.**

```
> 그 요건을 추가로 확정할게. 요구사항 정의서와 설계서에 반영해줘.
```

에이전트는 요구사항 정의서에 새 요건을 더하고, 연결된 설계서도 함께 고친 뒤 확인을 받습니다.

```
다음과 같이 반영하겠습니다.
  요구사항 정의서: R-11 재해 복구를 위한 리전 이중화 추가
  설계서: 두 리전에 서비스 이중 배치, 장애 시 자동 전환, 모니터링을 양쪽 리전으로 확장
  변경 사유: 재해 복구 요건 추가(고객 요청)
저장할까요? (y/n)
> y
저장했습니다. 변경 내용과 사유를 그래프에 함께 기록했습니다.
```

### 2) 받는 결과

먼저 영향 범위를 받아 어디까지 손대야 하는지 파악했고, 이어서 실제로 갱신된 요구사항 정의서(R-11 추가)와 설계서를 받았습니다. 무엇을 왜 바꿨는지가 함께 남아 나중에 추적할 수 있습니다.

## 시나리오 5.3 — 설계 질의응답

설계에 관해 궁금한 점을 자연어로 물어보고 정리된 답을 받습니다.

### 1) 입력하는 질문

```
> 운영 환경 노드마다 IP랑 CPU, 메모리, 디스크 크기를 정리해줘.
```

에이전트는 그래프에 쌓인 현행 시스템 분석서와 설계서에서 노드와 사양을 찾아 한 표로 모아 보여줍니다. (조회라서 확인을 받지 않습니다.)

```
운영 환경 노드 사양입니다.

  노드      IP            CPU       메모리    디스크
  core-01   10.0.1.21     vCPU 8    32GB     500GB
  core-02   10.0.1.22     vCPU 8    32GB     500GB
  db-01     10.0.2.21     vCPU 16   64GB     2TB

  ※ 클라우드 전환·MSA 분리로 새로 생기는 서비스 노드는 아직 IP가 배정되지 않았습니다.
```

### 2) 받는 결과

노드별 IP와 CPU, 메모리, 디스크 크기를 정리한 표를 한 번에 받았습니다. 여러 문서를 직접 뒤지지 않고 그래프에서 바로 모았기 때문에, 내용은 지금까지 쌓인 산출물과 일치합니다.

## 시나리오 5.4 — 산출물 출력

지금까지 작성한 산출물을 제출용 문서로 정리합니다.

### 1) 입력하는 명령

```
> 현재 산출물을 문서별로 HTML로 만들어줘.
```

에이전트는 그래프에 있는 각 산출물을 문서 단위로 모아 HTML 파일로 만들어 작업 폴더에 저장하고, 결과를 알려 줍니다.

```
다음 문서를 HTML로 만들었습니다.
  out/requirements.html   요구사항 정의서 (R-1 ~ R-11)
  out/current-system.html 현행 시스템 분석서
  out/design.html         아키텍처 설계서
각 문서의 내용은 그래프에서 가져왔기 때문에 최신 상태입니다.
```

### 2) 받는 결과

요구사항 정의서, 현행 시스템 분석서, 설계서가 각각 HTML 파일로 출력되었습니다. 앞선 시나리오에서 추가한 리전 이중화 요건(R-11)까지 반영되어, 출력물은 항상 그래프의 최신 상태와 같습니다.
