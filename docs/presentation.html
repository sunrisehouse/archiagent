Problem

문제 정의 — 산출물이 "문서"라서 생기는 일

#문제증상
P1최신성 붕괴시스템은 계속 배포·변경되는데 문서는 갱신이 늦음 → 설계와 실제의 괴리(drift)
P2추적 불가컴포넌트 간 의존성·영향 범위를 문서에서 추적 불가. "이거 바꾸면 뭐가 깨지지?"에 답 못함
P3의도-실제 단절설계 의도(why)는 문서에, 실제 운영 상태(what-is)는 인프라 콘솔에. 서로 연결 안 됨
P4질의 불가"현재 결제 도메인 구조 알려줘" 같은 자연어 질의·온보딩 설명을 매번 수작업

근본 원인 — 산출물이 연결된 구조(graph)가 아니라 납작한 텍스트(doc)로 저장되고, 실제 시스템 상태와 분리되어 있다.

Intent

기획 의도

① 문서 → 지식 그래프

컴포넌트·의존성·결정(ADR)을 노드·엣지로(Graph DB) + 자연어 의미검색(Vector DB). → P2·P4

② 설계 ↔ 실제 연계

설계를 아는 Agent A + 배포 상태를 아는 Agent B를 오케스트레이터로 묶어 "의도 vs 실제"를 교차 질의. → P1·P3

③ 반복 작업 보조

영향분석 · 현황 파악 · 온보딩 설명 같은 아키텍트의 반복 업무를 에이전트가 대신.

④ 시나리오 중심 PoC

완성형 제품이 아니라, 몇 가지 시나리오로 에이전트 간 연계의 가치를 시연.

주요 기능 · 시나리오 1

설계 vs 실제 Drift 탐지 — 연계의 핵심 가치

sequenceDiagram
  actor U as Architect
  participant O as Orchestrator
  participant A as Agent A
  participant B as Agent B
  U->>O: "결제 도메인 설계 대비
실제 배포 차이 알려줘"
  O->>A: 설계상 결제 토폴로지
  A-->>O: intended 구성
  O->>B: 실제 배포 컴포넌트
  B-->>O: observed 상태
  O->>O: 교차 비교 (drift)
  O-->>U: Drift 리포트

산출 — Drift 리포트

  • 누락 설계엔 있으나 미배포
  • 추가 배포됐으나 설계에 없음
  • 불일치 버전·의존성 차이

② Agent Runtime

Orchestration · Instruction

Orchestration 부문

  • SDK query() 루프 진입점
  • 의도 분해 → 시나리오 라우팅(drift/영향도/Q&A)
  • Agent A = subagent, Agent B = MCP tool
  • 정지 조건: 시나리오별 최대 반복 · 실패 시 graceful degrade
  • 비가역 작업(그래프 쓰기)은 AskUserQuestion 확인

Agent Instruction · Prompt

  • Orchestrator: 역할·라우팅 휴리스틱·"설계는 A, 실제는 B" 도구 규칙
  • Agent A: 그래프 데이터 모델·질의 패턴, "그래프에 없으면 없다고", concise|detailed
  • 설계 원칙은 CLAUDE.md §5(Harness Engineering) 준수

Graph DB (예: Neo4j)

설계 노드와 관측 노드(DeployedComponent)를 분리 저장 → drift 비교의 기반.

Vector DB (예: pgvector/Qdrant)

문서 청크·노드 설명 임베딩 → 자연어 질의를 그래프 진입점으로 변환.

Wrap-up

정리 — PoC 성공 기준 & 남은 결정

성공 기준

  • 3개 시나리오에서 오케스트레이터가 두 에이전트를 호출·조합
  • Drift 리포트가 누락/추가/불일치 3범주 구분
  • 영향도 분석이 다단계 의존성 + 실제 가동 우선순위
  • Agent B를 모킹 계약으로 대체 가능

남은 결정 (Open)

  • Graph/Vector DB 제품 선정
  • Front 형태(웹 챗 vs CLI)
  • Agent B 연계 계약 상세 스키마(외부 팀 합의)
  • 인제스트 대상 산출물 포맷 범위
