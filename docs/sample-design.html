1. 개요

본 설계서는 현행 온프레미스 시스템(core-01, core-02, db-01)을 클라우드 기반 마이크로서비스 구조로 전환하는 목표 아키텍처를 정의한다. 전환 후에도 금융 규제 준수, 고가용성, 무중단 운영을 보장하는 것을 핵심 설계 원칙으로 삼는다.

3. 목표 아키텍처

3.1 클라우드 전환 전략 (→ R-1)

  • 리프트 앤 리아키텍트(Lift-and-Re-architect) 방식을 채택한다. 단순 VM 이전이 아니라, 클라우드 네이티브 컨테이너·관리형 서비스로 재구성하여 클라우드 이점을 최대화한다.
  • 퍼블릭 클라우드(AWS / Azure / GCP 중 고객 정책에 따라 확정)의 멀티 가용 영역(Multi-AZ) 구성을 기본으로 한다.
  • 온프레미스 DB(db-01, 2TB)는 클라우드 관리형 RDB(예: Amazon RDS / Azure SQL)로 마이그레이션하고, 초기 병행 운영 구간에 데이터 동기화 채널(DMS)을 유지한다.

핵심 업무 기능을 다음 6개 도메인 서비스로 분해한다. (세부 경계는 도메인 분석 후 확정)

  • 각 서비스는 독립 컨테이너 이미지로 패키징하고, 컨테이너 오케스트레이터(Kubernetes)에서 서비스별 `Deployment` 단위로 분리 운영한다.

3.3 고가용성 설계 (→ R-4, R-6)

  • 컨트롤 플레인: Kubernetes 클러스터를 3개 가용 영역(AZ)에 걸쳐 구성한다. 마스터 노드 3중화.
  • 워크로드 노드: 각 서비스 파드는 최소 2개 레플리카를 유지하며, `PodAntiAffinity`로 단일 AZ 장애 시에도 서비스가 유지되도록 분산 배치한다.
  • 데이터베이스: Primary + Read Replica 구성(Multi-AZ 동기 복제). 장애 시 자동 Failover(RTO ≤ 30초 목표).
  • API Gateway / 로드밸런서: L7 로드밸런서를 서비스 진입점에 배치하고, 헬스체크 기반 자동 트래픽 절체로 단일 인스턴스 장애를 무시한다.
  • 회로 차단기(Circuit Breaker): 서비스 간 호출에 Circuit Breaker 패턴(예: Resilience4j)을 적용하여 장애 전파를 차단한다.

3.4 성능·확장성 설계 (→ R-6, R-8)

  • 수평 자동 확장(HPA): 각 서비스별로 CPU 사용률 또는 커스텀 메트릭(TPS) 기준 HPA를 설정한다. 거래 서비스는 피크 대비 최대 10× 스케일아웃을 목표로 한다.
  • 캐시 계층: 고객 정보·상품 정보처럼 읽기 빈도가 높은 데이터는 분산 캐시(Redis Cluster)를 도입하여 DB 부하를 경감한다.
  • CDN: 정적 리소스는 CDN을 통해 제공하여 엣지 응답 속도를 확보한다.
  • 피크 시 응답 목표: P99 응답 시간 ≤ 500 ms (거래 서비스 기준, 별도 SLA 협의 후 확정).

3.5 보안 설계 (→ R-5)

  • 네트워크 분리: 퍼블릭 서브넷(API Gateway·CDN), 프라이빗 서브넷(서비스 파드), 격리 서브넷(DB)으로 3-tier 분리. 서브넷 간 통신은 Security Group / Network Policy로 최소 권한 허용.
  • 전송 암호화: 모든 외부·내부 통신에 TLS 1.2 이상 필수 적용. 내부 서비스 간 mTLS 적용(Service Mesh, 예: Istio).
  • 저장 암호화: DB 및 오브젝트 스토리지는 관리형 KMS 키로 암호화.
  • 비밀 관리: DB 자격증명·API 키는 Secrets Manager(예: AWS Secrets Manager / HashiCorp Vault)에서 관리하며, 소스 코드·환경변수에 하드코딩 금지.
  • 인증/인가: OAuth 2.0 + OIDC 기반 토큰 인증. 역할 기반 접근 제어(RBAC) 적용.
  • 취약점 관리: 컨테이너 이미지 빌드 파이프라인에 CVE 스캔(예: Trivy) 통합. 고위험 취약점 존재 시 배포 차단.
  • 금융 보안 규제(전자금융감독규정 등) 요건은 보안 아키텍처 세부 설계 단계에서 항목별 대응 매핑표를 별도 작성한다.

3.6 모니터링 및 알림 설계 (→ R-3, R-7)

  • 메트릭 수집: Prometheus를 각 서비스 사이드카 또는 exporter로 배포하여 CPU·메모리·TPS·에러율·응답 시간을 수집한다.
  • 로그 수집: Fluentd / Fluent Bit → Elasticsearch(OpenSearch) 파이프라인으로 전 서비스 로그를 중앙화한다. Kibana / OpenSearch Dashboards로 실시간 조회.
  • 분산 추적: Jaeger / Zipkin으로 서비스 간 요청 흐름을 추적하여 병목 구간을 신속하게 식별한다.
  • 대시보드: Grafana에 서비스별·클러스터 전체 통합 대시보드를 구성한다.
  • 알림 정책: Alertmanager로 임계치 위반 시 운영팀 Slack·이메일·PagerDuty에 즉시 발송. 주요 알림 규칙 예시:
  • 서비스 에러율 > 1% (5분 지속) → 경고
  • 서비스 에러율 > 5% (2분 지속) → 긴급
  • 파드 재시작 > 3회/10분 → 경고
  • P99 응답 > 1초 (5분 지속) → 경고

3.7 감사 로그 설계 (→ R-10)

  • 모든 서비스는 구조화된 감사 로그(JSON 포맷)를 비동기 메시지 큐(Kafka)를 통해 감사 로그 서비스로 전송한다. 호출 서비스의 성능에 영향을 주지 않는다.
  • 감사 로그에는 다음 필드를 필수 포함한다: `timestamp`, `traceId`, `userId`, `action`, `targetResource`, `sourceIp`, `result`, `detail`.
  • 감사 로그 저장소는 불변 스토리지(WORM 정책 적용 오브젝트 스토리지)에 장기 보관(규제 요건에 따라 최소 5년)한다.
  • 로그 위변조 방지를 위해 저장 시 해시 체인 또는 타임스탬프 서명을 적용한다.

3.8 무중단 배포 설계 (→ R-9)

  • 배포 전략: 기본 전략은 롤링 업데이트(Rolling Update). 위험도가 높은 거래 서비스는 카나리 배포(Canary Release) 를 적용하여 소수 트래픽으로 사전 검증 후 전체 전환한다.
  • CI/CD 파이프라인(예: GitLab CI / GitHub Actions → ArgoCD)에서 자동 빌드·테스트·이미지 스캔·배포를 순차 수행한다.
  • 배포 실패 시 ArgoCD의 자동 롤백 기능으로 이전 버전으로 즉시 복구한다.
  • DB 스키마 변경은 전진 호환 마이그레이션(Expand-Contract 패턴) 으로 적용하여 배포 중 서비스 중단을 방지한다.
